## 🏗️ 最终架构（数据流）

```
用户对着麦克风说话
     ↓
[VoiceDetector] VAD 检测到音量超过阈值
     ↓ onRecordingStart()
[page.tsx] 立即暂停播客
     ↓
[VoiceDetector] 录音中... 静默 1.5 秒后自动停止
     ↓ onRecordingComplete(blob)
[/api/transcribe] → ffmpeg 转 PCM → 讯飞 WebSocket → 识别文本
     ↓
[/api/ask] 文本 + 实时上下文 → DeepSeek → 简短回答
     ↓
[useTTS] 浏览器 SpeechSynthesis 语音播报（1.2x 语速）
     ↓ onEnd()
[page.tsx] 自动恢复播客播放

同时在后台：
[useLiveTranscript] 每 30 秒截取播客音频 → Groq Whisper 转写
                   → 滑动窗口保留最近 3 分钟 → 作为 AI 回答的上下文
```

## 🧠 关键技术决策（为什么选 A 不选 B）

### 1. ASR：讯飞 vs Groq Whisper vs 浏览器 Web Speech API

| 方案 | 试过了吗 | 结果 |
|------|---------|------|
| Groq Whisper (SDK) | ✅ | Node.js 24 的 fetch/FormData 发 multipart 给 Groq 触发 Cloudflare 403。curl 能用但 SDK 不行 |
| Groq Whisper (curl) | ✅ | curl 能通，但频繁调用会 Internal Server Error |
| 浏览器 Web Speech API | ✅ | 能跑通，但延迟高（等 isFinal 慢）、中英混合差（"AutoML" → "奥特曼"）|
| **讯飞 WebSocket** | ✅ | **最终选择**。国内服务器无网络问题，中英混合好，速度快 |

**核心教训**：Node.js 24 的原生 fetch 和第三方 API 的 Cloudflare 防护不兼容，这不是代码 bug，是环境问题。遇到 403 先用 curl 验证 API 本身是否正常。

### 2. TTS：浏览器 SpeechSynthesis vs API TTS

选了浏览器内置：零延迟、零成本、可控语速/音调。demo 够用。
如果面试官觉得声音太机械，可以说"生产环境会换 API TTS（如讯飞/Azure）"。

### 3. VAD：音量检测 vs Web Speech API onspeechstart

最终用 Web Audio API AnalyserNode 做音量检测。阈值 0.08，静默超时 1.5 秒。
echoCancellation/noiseSuppression 已开启，但外放场景下播客声音仍可能触发误检。
**戴耳机效果好很多**，这是设计的目标场景（通勤/跑步戴耳机听播客）。

### 4. 实时上下文：硬编码 vs 真实转写

从硬编码 `DEMO_CONTEXT` 改成了真实的滑动窗口转写。每 30 秒截一段播客音频发 Groq Whisper，保留最近 6 段（约 3 分钟）。AI 回答时拿到的是用户当前真正在听的内容。

## 🐛 踩坑手册

### 1. Groq SDK 403 Forbidden
- **问题**：Node.js 24 环境下，groq-sdk 和原生 fetch/FormData 发 multipart 给 Groq API 全部返回 403
- **原因**：Cloudflare 的 TLS fingerprint 检测，Node.js 的 TLS 指纹和浏览器/curl 不同，被拦截
- **排查过程**：curl 直接调 API → 200 成功 → 确认不是 API key 问题 → 确认是 Node.js 发请求的方式
- **解决**：用 `execFileSync("curl", args)` 绕过，最终换成讯飞 API 彻底解决
- **以后怎么避免**：遇到 API 403，先用 curl 验证。如果 curl 能通但代码不行，是 HTTP client 的问题

### 2. VAD 误触（播客外放被麦克风拾取）
- **问题**：电脑外放播客时，麦克风收到播客声音，VAD 误判为用户在说话
- **原因**：音量阈值太低（0.03），播客声音就超过了
- **解决**：阈值提高到 0.08 + 最短录音 800ms + 空文本跳过逻辑
- **根本解决**：目标场景是戴耳机听播客，麦克风只收到用户声音

### 3. ffmpeg-static 路径错误
- **问题**：Next.js 运行时 `import ffmpegPath from "ffmpeg-static"` 返回 `\ROOT\node_modules\...`
- **原因**：Next.js 的模块解析和 Node.js 不同
- **解决**：用 `path.resolve(process.cwd(), "node_modules", "ffmpeg-static", "ffmpeg.exe")` 拼绝对路径

### 4. 讯飞 WebSocket 发送太慢（20 秒延迟）
- **问题**：按实时速度（40ms 一帧）发音频，3 秒的录音要 3 秒发完
- **解决**：去掉 setTimeout 间隔，所有帧一次性快速发完。讯飞服务端能处理

### 5. 讯飞 API Key/Secret 搞反
- **问题**：401 Unauthorized
- **原因**：APIKey 和 APISecret 的值放反了
- **教训**：讯飞的命名是 APIKey = 用于 authorization 里的 api_key 字段，APISecret = 用于 HMAC 签名

## 💡 今天最值得记住的 3 件事

1. **Node.js 发 HTTP 请求可能被 Cloudflare 拦截**——同样的 API，curl 能通但 Node.js fetch/SDK 被 403。遇到这种情况别死磕代码，先换调用方式（curl/其他 HTTP client），或者换国内 API。

2. **VAD 音量检测 vs 语音识别 API 检测是两回事**——音量检测快但分不清谁的声音；语音识别 API（如 Web Speech API）能分但慢。最终选了音量检测 + 提高阈值 + 目标场景是戴耳机（从根本上解决）。

3. **先跑通再优化**——从 Groq 403 → curl 绕过 → Web Speech API → 讯飞，每次卡住不死磕，换方案。最终讯飞一次性解决了速度和准确度两个问题。这就是工程思维：目标是跑通，不是证明某个技术方案能用。
