## 🧱 基础语句：今天碰到了什么语法

### useRef vs useState — 什么时候用哪个
```typescript
// ref：值变了不触发重新渲染，回调函数里永远读到最新值
const confirmingRef = useRef(false);

// state：值变了触发重新渲染，但回调函数闭包可能捕获旧值
const [confirmCountdown, setConfirmCountdown] = useState<number | null>(null);
```
类比：ref 像白板上的字，谁来看都是最新的；state 像拍的照片，拍完那一刻就定格了，回调函数拿着的是"拍照那一刻"的值。

今天在 Step 5 主动确认流程里踩了这个坑：TTS 的 onEnd 回调在闭包里执行，如果用 state 存 confirming 状态，读到的永远是创建闭包时的旧值。改成 ref 才正确。

### useCallback 依赖数组
```typescript
const handleRecordingComplete = useCallback(async (audioBlob: Blob) => {
  // ... 用到了 getContext, speak, addLog, silenceOverride
}, [getContext, speak, addLog, silenceOverride]); // ← 用到谁就写谁
```
类比：依赖数组就像食材清单——你做菜用了什么原料就要写上去，不然下次原料换了你还在用旧的。

### forwardRef + useImperativeHandle — 让父组件控制子组件
```typescript
// AudioPlayer 暴露 pause/resume/fadeVolume 给父组件
useImperativeHandle(ref, () => ({
  pause: () => audioRef.current?.pause(),
  fadeVolume: (target, duration) => { /* requestAnimationFrame 插值 */ },
}));
```
类比：子组件是一台机器，useImperativeHandle 是给机器装上遥控器按钮，父组件拿着遥控器就能操作。

## 🏗️ 架构和信息流：一个请求从头到尾经历了什么

```
用户说话
  │
  ▼
VoiceDetector（VAD 音量检测）
  │ 音量 > 0.08 → 开始录音，通知 page.tsx 暂停播客
  │ 静默 2.5 秒 → 停止录音
  ▼
page.tsx handleRecordingComplete
  │
  ├─ ASR 转写（讯飞 WebSocket，~500ms）
  │   ▼
  │ 文本分流判断：
  │   ├─ 空/语气词/喷嚏  → 非语义信号 → 恢复播客 + 音量渐降
  │   ├─ 重复字符 > 40%   → ASR 乱码 → 恢复播客 + 提示重说
  │   ├─ 包含"等一下"     → 续说信号 → 延长超时到 5 秒，等用户继续
  │   └─ 正常文本         → 继续 ↓
  │
  ├─ 获取播客上下文（6段加权窗口）
  │
  ├─ DeepSeek LLM（~1200ms）
  │   ▼
  │ 回复分流判断：
  │   ├─ 包含"你是想问" → 确认模式 → TTS 播完后等 5 秒
  │   └─ 正常回答       → 暂存日志
  │
  ├─ TTS 播报（浏览器 SpeechSynthesis）
  │   ▼
  │ TTS 结束：
  │   ├─ 确认模式 → 5 秒倒计时，等用户回应或超时恢复
  │   └─ 正常模式 → 写日志 → 恢复播客
  │
  └─ 端到端 ~2000ms
```

## 🧠 工程思维：今天做了什么技术决策

### 决策 1：非语义检测用 ASR 文本匹配，不用频谱分析
- **选了**：ASR 返回后做文本层判断（空文本/语气词列表/重复率检测）
- **没选**：浏览器端 Web Audio API 做频谱分析区分咳嗽和说话
- **为什么**：频谱分析在不同麦克风、环境噪音下表现差异大，不稳定。ASR 本身已经在做语音识别了，利用它的结果做判断，零额外开销且更可靠

### 决策 2：确认状态用 ref 不用 state
- **问题**：TTS onEnd 回调在闭包里，读不到最新的 state
- **方案**：用 useRef 存 confirming 状态，ref.current 永远是最新值
- **教训**：React Hooks 最常见的坑——闭包捕获旧值。凡是在异步回调/定时器里需要读取最新值的，都用 ref

### 决策 3：wasPlayingRef 改成基于业务语义
- **问题**：连续提问后播客不恢复，因为 `audio.isPlaying()` 在 `resume()` 还没 resolve 时返回 false
- **修复**：改成 `wasPlayingRef = !!audioFile`（有音频就恢复），用业务语义代替底层状态
- **思维方式**：底层 API 的异步行为不可控时，退一步用更高层的判断条件

## 🐛 踩坑记录

### 坑 1：静默超时 1.5 秒太短
- **问题**：用户说"等一下..."还没说完就被截断
- **原因**：自然说话中间喘气/思考 1-2 秒很正常，1.5 秒太激进
- **怎么修的**：调到 2.5 秒 + 续说信号词检测（"等一下" → 延长到 5 秒）
- **下次怎么避免**：语音相关的超时参数，先用真人测几次再定，不要拍脑袋

### 坑 2：ASR 乱码穿透
- **问题**："是是我是我是是我是我是" 被当成正常问题送给 LLM
- **原因**：只检查了空文本和语气词，没考虑 ASR 产生的垃圾
- **怎么修的**：加 `isGarbledASR()` 函数，单字重复率 > 40% 判定为乱码
- **下次怎么避免**：ASR 的输出永远不可信，做防御性检查

### 坑 3：打喷嚏后播客卡住
- **问题**：喷嚏触发 VAD → ASR 返回"阿七" → 不在词表里 → 没恢复播客
- **原因**：乱码检测分支 `return` 了但没恢复播客
- **怎么修的**：每个 early return 分支都检查是否需要恢复播客
- **下次怎么避免**：每个分支都问自己"退出时系统该是什么状态"

### 坑 4：连续提问后不恢复播客
- **问题**：第一个问题答完恢复播客，立刻问第二个，第二个答完后播客不恢复了
- **原因**：`resume()` 是异步的，`isPlaying()` 还没来得及返回 true
- **怎么修的**：用 `!!audioFile` 代替 `isPlaying()` 判断
- **下次怎么避免**：异步 API 的返回值不要立刻用来做判断

## 💡 今天最值得记住的 3 件事

1. **ref vs state 的选择不是随意的**：异步回调/定时器里要读最新值 → 用 ref；需要触发 UI 更新 → 用 state。搞反了就会出"闭包捕获旧值"的 bug，而且很难 debug。

2. **每个 early return 都要问"系统该恢复到什么状态"**：打喷嚏卡住就是因为一个分支 return 了但忘了恢复播客。养成习惯：写 return 之前想一想"我打断了正常流程，之前被改变的状态都还原了吗"。

3. **"在正确的层做事情"比"用更酷的技术"重要**：非语义检测可以做频谱分析（听起来很高级），但在 ASR 文本层做匹配更简单、更稳定、零额外开销。工程决策不是选最炫的，是选最稳的。
